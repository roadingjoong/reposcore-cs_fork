# 이슈 작업 선점 관련 규정 가이드

## 1. 작업 선점 규칙
- 새로운 이슈가 등록된 후 **24시간 이내**에  
  이슈 작성자를 제외한 다른 사람이 아래와 같은 댓글을 작성하여 작업을 선점할 수 있습니다.
  - 예: `제가 하겠습니다`

- **동시 선점 이슈 갯수 제한**  
  동시에 선점 가능한 이슈는기능 1개, 문서 1개로 제한됩니다.

- **24시간 이내에 자원자가 없는 경우**,  
  이슈를 등록한 본인이 댓글을 통해 해당 작업을 맡을 수 있습니다.

- **작업 후 PR 제출 시 새로운 이슈 선점 가능**  
  머지되지 않은 PR이 2개 쌓일 시 새로운 이슈 선점 불가(문서2개, 기능/버그2개, 최대 4개까지 쌓아놓고 기다릴 수 있음)


---

## 2. 작업 완료 기한

작업을 선점(할당)받은 후에는 아래 기한 내에 PR(Pull Request)을 제출해야 합니다.

| 작업 유형 | 기한 (선점 댓글 작성 시점 기준) |
| :--- | :--- |
| 📝 **문서 작업** (오타 수정, README 보완 등) | **24시간 이내** |
| 💻 **코드 작업** (기능 개발, 버그 수정 등) | **48시간 이내** |

> **⚠️ 주의:** 기한 내에 작업이 완료되지 않으면 다른 기여자를 위해 해당 이슈의 선점이 해제됩니다.

---

## 3. 기타 사항
- 기한 내 작업 **미진행** 또는 **작업 포기** 시,
  다른 사람이 해당 이슈를 다시 선점할 수 있습니다.
  - 개인 사정으로 진행이 어려울 경우 즉시 댓글로 **포기 의사**를 남겨야합니다.
  - 포기 의사를 표현했을 시 **이슈 작성자**는 이슈 작성 시간으로부터 **24시간 이후** 작업 선전할 권한을 얻습니다. (24시간 이전에는 권한 없음)
  - 기한(24시간/48시간) 초과 후 재선점이 가능하려면, 해당 이슈에 연결된 머지 대기 중인 PR이 없어야 합니다.
   - 이미 PR이 제출되어 머지를 기다리는 중이라면, 기한이 지나도 다른 사람이 재선점할 수 없습니다.
   - PR이 닫히거나(Close) 머지가 완료된 이후에야 재선점이 허용됩니다.
  - 포기 의사를 표현했을 시 **이슈 작성자**는 이슈 작성 시간으로부터 **24시간 이후** 작업 선점할 권한을 얻습니다. (24시간 이전에는 권한 없음)
- 작업 시작 시 빠른 진행을 위해 관련 내용을 충분히 확인한 후 선점합니다.
- 어떤 이슈에 대한 것인지 #이슈번호 본문 처음에 작성
- 어떤 작업 내용인지 되도록 간단히 요약한 내용 작성

---

## 4. 이슈 작성 시 주의사항
- 버전 정보 명시: 코드 수정이 빈번하므로 반드시 작업 기준이 되는 버전를 포함해야 합니다.
- **버전 정보 명시:** 코드 수정이 빈번하므로 반드시 작업 기준이 되는 버전(Commit ID)을 포함해야 합니다.
  1. **Commit ID 확인:** GitHub 레포지토리의 [Commits] 탭에서 확인 가능한 7자리 해시값(예: `74785f8`)을 입력합니다.
  2. **버전 링크 복사:** 해당 커밋 상세 페이지의 URL 주소를 붙여넣습니다. (예: URL 붙여 넣으면 preview 했를을 때[74785f8] 이렇게 링크가 뜹니다. )
- 구체적인 예시 제시: 추상적인 설명("유의", "주의") 대신, 수정 전/후의 모습이나 구체적인 마크다운 문구 예시를 제시해야 합니다.
  "성능에 유의해주세요.", "에러가 발생하지 않도록 해주세요." 와 같은 말들은 무엇이 성능 문제 인지, 어떤 에러를 말하는지 불명확합니다.
  "대량 데이터 조회 시 N+1 쿼리가 발생할 수 있으므로 `fetch join`을 사용해주세요." , "null 값이 들어올 경우 `NullPointerException`이 발생하므로 기본값 처리 또는 Optional을 사용해주세요." 와 같이
  문제가 발생하는 **구체적 상황을 명시**하고 발생 가능 **에러 또는 이슈를 명확화**해야 합니다.
- 정확한 파일 경로: 단순히 파일명만 적지 않고, main/docs/README.md와 같이 전체 경로를 명시해야 합니다.
- 작성 시 포함해야 할 내용들 :
  **언제 (상황)**
  -어떤 조건에서 문제가 발생하는지
  **무엇이 문제인지 (이슈)**
  발생 가능한 오류 / 성능 문제 / 사이드 이펙트
  **어떻게 해결할지 (가이드)**
  구체적 구현 방법 또는 권장 방식

## 5. PR 작성 시 주의사항
**타이틀 작성**
- PR의 목적이 한눈에 드러나도록 구체적으로 작성해야 합니다.
예) "수정" (X) "로그인 API NullPointerException 수정" (O)

**PR 본문 작성**
- 어떤 변경을 했는지 명확하게 작성해야 합니다.
- 변경 전 / 변경 후를 비교할 수 있도록 작성합니다.

**이슈 번호 작성 및 방법**
>- PR과 이슈 자동 연동 (GitHub Auto-linking) 기능을 이용하기 위해서, PR 본문(Description)에 GitHub에서 지원하는 지정된 키워드와 이슈 번호를 함께 작성해야합니다.
      
- 지정된 키워드 예시
>
>Closes #123
>
>Fixes #42
>
>Resolves #7

- 원리
> PR이 병합(Merge)되었을 때 연관된 이슈가 자동으로 닫히도록 설정하여, 불필요한 관리 부담을 줄이고 이슈 누락을 방지합니다.
> 
> 실행 조건: PR이 저장소의 기본 브랜치(예: main)에 완전히 병합되는 순간 작동합니다.

- 예외 케이스 (자동 연동이 동작하지 않는 경우)
>다른 저장소 이슈 연동: 단일 이슈 번호(#123) 대신 사용자명/저장소명#이슈번호 형태로 명시해야 합니다. (예: Closes owner/repo#123)
>
>Draft PR 상태: PR이 임시(Draft) 상태일 때는 병합할 수 없으므로 이슈가 닫히지 않습니다.
>
>기본 브랜치가 아닌 경우: main이 아닌 테스트용 기능 브랜치 등으로 PR이 머지될 때는 자동 종료 트리거가 작동하지 않습니다.
